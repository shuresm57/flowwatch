## 1. Problem Description

**What problem does Flowwatch solve?**

Modern networks generate enormous volumes of traffic. Detecting malicious activity (DDoS attacks, port scans, brute-force logins, web exploits) manually is impossible at scale. Flowwatch is a real-time network intrusion detection system (NIDS) that classifies network traffic flows as benign or as one of 14 known attack types using a trained machine learning model.

**Why machine learning?**
Traditional rule-based intrusion detection (e.g. Snort signatures) requires manual rule updates and misses novel attack patterns. A supervised ML model learns statistical patterns across thousands of features automatically and can generalize to new variants.

**Scope:**
- Input: 78 numerical features describing a single network flow (packet sizes, inter-arrival times, flag counts, byte ratios, etc.)
- Output: a class label + confidence score (e.g. `DDoS`, `0.998`)

## 2. Dataset

**Source:** [CICIDS2017 — Canadian Institute for Cybersecurity](https://www.kaggle.com/datasets/cicdataset/cicids2017) (available on Kaggle)

**What it contains:**
- Network traffic captured over 5 days in a controlled lab environment
- Both normal (benign) and attack traffic, generated by a realistic traffic profile
- 8 CSV files, ~3 million rows total
- 78 numerical features extracted by CICFlowMeter from raw pcap files

**Class distribution (after cleaning):**

| Class | Approx. samples |
|-------|----------------|
| BENIGN | ~2,000,000 |
| DoS Hulk | ~231,000 |
| PortScan | ~159,000 |
| DDoS | ~128,000 |
| DoS GoldenEye | ~10,000 |
| FTP-Patator | ~8,000 |
| SSH-Patator | ~6,000 |
| DoS slowloris | ~5,800 |
| DoS Slowhttptest | ~5,500 |

| Web Attack — SQL Injection | ~21 |

**Key challenge:** Severe class imbalance — benign traffic outnumbers the rarest attack class by a factor of ~180,000x.

## 3. Data Preparation

**Notebook:** `training/preprocess.ipynb`

Steps performed:

1. **Load all 8 CSV files** into a single pandas DataFrame (~3M rows)
2. **Strip whitespace from column names** (the raw CSVs have leading spaces in headers)
3. **Remove duplicate rows** — reduces noise from duplicate flow records
4. **Drop rows with `inf` or `NaN` values** — a small number of flows have infinite feature values (division by zero in flow duration calculations); these are dropped
5. **Label-encode the target column** (`Label`) — maps string class names to integers 0–14 and saves the mapping to `metadata.json`
6. **Save cleaned data** to `cleaned.csv` — 2,520,798 rows with 78 features + 1 target column

The `metadata.json` file produced here is also loaded by the Node.js API at runtime so predictions can be decoded back to human-readable class names.

## 4. Model Training

**Notebook:** `training/xgboost_ids.ipynb`

### Algorithm choice: XGBoost
XGBoost (Extreme Gradient Boosting) is a well-established algorithm for tabular classification tasks. It handles:
- Mixed feature scales without normalization
- Non-linear decision boundaries
- Feature importance out of the box

A deep neural network was considered but XGBoost was chosen because the features are already engineered (not raw packets), the dataset is tabular, and XGBoost trains significantly faster on CPU.

### Pipeline

1. **Train/test split** — 80% train, 20% test, stratified by class to preserve rare-class proportions

2. **Class weighting** — compute per-class sample weights inversely proportional to class frequency. This forces the model to pay attention to rare attack types (e.g. Heartbleed with only 11 samples)

3. **Baseline model** — train an XGBoost classifier with default hyperparameters to establish a performance baseline (99.49% accuracy)

4. **Hyperparameter tuning with Optuna** — run 10 trials on a 15% stratified subset to speed up search. Optuna uses Tree-structured Parzen Estimator (TPE) to search efficiently over:
   - `n_estimators`: 100–500
   - `max_depth`: 3–10
   - `learning_rate`: 0.01–0.3
   - `subsample`, `colsample_bytree`: 0.5–1.0
   - `gamma`, `reg_lambda`, `reg_alpha`: regularization

5. **Final model** — train on the full training set using Optuna's best parameters:
   ```
   n_estimators=249, max_depth=3, learning_rate=0.129,
   subsample=0.872, colsample_bytree=0.932
   ```

6. **Export to ONNX** — convert the trained model to the ONNX format using `skl2onnx` / `onnxmltools`. This allows the model to be loaded in Node.js via `onnxruntime-node` without any Python dependency at inference time.

## 5. Model Evaluation

**Results on the held-out 20% test set (504,159 flows):**

| Metric | Value |
|--------|-------|
| Accuracy | **99.79%** |
| ROC AUC (macro OvR) | **0.999** |
| F1 Score (macro avg) | **0.8591** |
| F1 Score (weighted avg) | ~0.999 |

**Why macro F1 matters here:**
Accuracy alone is misleading when classes are imbalanced. A model that always predicts BENIGN would score ~79% accuracy but be useless. Macro F1 weights all classes equally, penalizing poor performance on rare attack types.

**Observations:**
- Excellent performance on high-frequency classes (BENIGN, DDoS, DoS Hulk, PortScan)
- Lower recall on the rarest classes (Heartbleed: 11 samples total, Infiltration: 36 samples) — expected given the extreme imbalance
- Feature importance analysis shows flow duration, packet length statistics, and inter-arrival time features are most predictive

**Visualizations produced in the notebook:**
- Confusion matrix (normalized)
- Per-class ROC curves
- Feature importance bar chart (top 20 features)
- t-SNE 2D projection of a stratified sample colored by class

## 6. Extras

### Published to a web server

The model is served via a lightweight Node.js/Express REST API (`src/`). No Python is required at inference time — the ONNX runtime handles model execution natively.

**API endpoints:**
| Endpoint | Description |
|----------|-------------|
| `GET /health` | Liveness check |
| `GET /features` | Returns feature names and class labels |
| `POST /analyze` | Classify a single flow (JSON body) |
| `POST /analyze/csv` | Batch-classify flows from CSV |

**Example request:**
```bash
curl -X POST http://localhost:3000/analyze \
  -H "Content-Type: application/json" \
  -d '{"features": [0.0, 1.5, 200.0, ...]}'
```

**Example response:**
```json
{
  "label": "DDoS",
  "label_id": 2,
  "confidence": 0.9987,
  "probabilities": {
    "BENIGN": 0.0001,
    "DDoS": 0.9987,
    "PortScan": 0.0005,
    ...
  }
}
```

### Deployment
The API is designed to run as a `systemctl` service on a Linux server. See `ROADMAP.md` for the full deployment guide including the systemd unit file.

### Cross-platform model format
Exporting to ONNX decouples the trained model from the Python/scikit-learn/XGBoost ecosystem. The same `.onnx` file could be loaded in C++, Java, Rust, or any other language with an ONNX runtime.
